How it works

Each rule edits one header on matching traffic, using the browser's declarativeNetRequest engine — edits are applied natively, even before the extension's code runs.

  • Request / Response — which direction of the exchange to edit.
  • Set replaces (or adds) the header, Remove strips it, Append adds a value alongside an existing one. Browsers only allow append on a small set of request headers (Accept, Accept-Language, Cache-Control, Cookie, …); response headers can generally be appended freely.
  • Scope — comma-separated domains; subdomains are included automatically (example.com also covers api.example.com). Leave empty to apply everywhere.
  • Advanced ⚙ — choose whether the scope matches the target host (the server being contacted) or the page host (the site the request originates from), and optionally add a URL filter such as ||example.com/api/*.
  • Order matters — when two rules touch the same header, the one higher in the list wins. Reorder with ↑ ↓.

Notes

  • Some headers are browser-reserved (e.g. Host) and can't be edited; if a rule is rejected you'll see an error toast here.
  • Unlike NoRefer v1, header names can't be regular expressions — that capability didn't survive the move from the deprecated blocking webRequest API. URL filters fill part of the gap.
  • Everything is stored locally. No telemetry, no sync, no servers.
